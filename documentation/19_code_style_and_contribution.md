## Назначение документа

Этот документ устанавливает единые правила написания кода, именования, документирования и внесения изменений в проект. Соблюдение этих правил обязательно для всех участников разработки.

## Автоматическое форматирование

Вопросы форматирования кода не обсуждаются — они решены автоматически:

- **black** форматирует весь Python-код. Максимальная длина строки: 100 символов.
- **ruff** проверяет стиль, неиспользуемые импорты, очевидные ошибки.
- **mypy** проверяет аннотации типов.

Запуск всех проверок:

```bash
black .
ruff check .
mypy iva/
```

Хук `pre-commit` запускает эти инструменты автоматически перед каждым коммитом. Коммит не может быть создан, если хотя бы одна проверка не пройдена.

## Именование

### Файлы и директории

Только строчные буквы, слова разделяются подчёркиванием. Имя файла соответствует основному классу или функции, которую он содержит.

```
psd_calculator.py       # правильно
PSDCalculator.py        # неправильно
psdcalc.py             # неправильно: сокращение ухудшает понимание
```

### Классы

Стиль CamelCase. Имя описывает что это, а не что оно делает.

```python
class SpectralPeak:         # правильно
class CalculateSpectrum:    # неправильно: это действие, не вещь
```

### Функции и методы

Стиль snake_case. Имя начинается с глагола и описывает действие.

```python
def calculate_reynolds_number(V, D, nu):    # правильно
def reynolds(V, D, nu):                     # неправильно: не ясно, что возвращает
def get_re(V, D, nu):                       # неправильно: сокращение
```

### Переменные

Имя объясняет содержимое, а не тип.

```python
sampling_rate_hz = 1000.0        # правильно
sr = 1000.0                      # неправильно: сокращение
float_value = 1000.0             # неправильно: тип не несёт смысла
```

Константы — все заглавные буквы:

```python
MAPE_DENOMINATOR_THRESHOLD = 1e-6
DEFAULT_WINDOW_SAMPLES = 1024
```

## Аннотации типов

Аннотации типов обязательны для всех функций во всех модулях `core/` и `app/`. В `ui/` — рекомендованы.

```python
def calculate_reynolds_number(
    mean_velocity_ms: float,
    diameter_m: float,
    kinematic_viscosity_m2s: float
) -> float:
    ...
```

Для NumPy-массивов используется `np.ndarray`. При необходимости уточнить форму или тип — в документационной строке.

## Документационные строки

Каждый публичный класс и каждая публичная функция имеет документационную строку. Она отвечает на вопрос «зачем», а не «как».

```python
def remove_mean(signal: np.ndarray) -> np.ndarray:
    """
    Удаляет постоянную составляющую из сигнала.

    Вычитает арифметическое среднее, что устраняет нулевую частоту
    в спектре и предотвращает искажение оценки RMS.

    Args:
        signal: одномерный массив значений сигнала.

    Returns:
        Массив той же формы с нулевым средним значением.
    """
    return signal - np.mean(signal)
```

Документационная строка не пересказывает очевидный код. Следующий вариант бесполезен:

```python
def remove_mean(signal):
    """Вычитает среднее из сигнала."""  # Это видно из названия функции
    return signal - np.mean(signal)
```

## Магические значения

Все числовые константы, не являющиеся очевидными (не 0, 1, -1, 2), должны быть именованными.

```python
# Правильно
OUTLIER_SCALE_FACTOR = 1.4826   # Делает MAD состоятельной оценкой σ для нормального распределения
sigma_robust = OUTLIER_SCALE_FACTOR * mad

# Неправильно
sigma_robust = 1.4826 * mad     # Магическое число без объяснения
```

## Размер функций и файлов

Функция выполняет одно действие и называется одним глаголом. Если функцию сложно назвать одним глаголом — она делает слишком много.

Рекомендуемый размер функции: до 30 строк содержательного кода. Превышение этого порога — сигнал к рефакторингу.

Рекомендуемый размер файла: до 200 строк. Если файл разрастается — часть логики следует вынести в отдельный модуль.

## Комментарии

Комментарии объясняют причину нетривиального решения, а не пересказывают код.

```python
# Применяем filtfilt вместо lfilter: двойной проход даёт нулевой фазовый сдвиг,
# что критично для правильного определения временны́х меток событий в сигнале.
filtered = scipy.signal.filtfilt(b, a, signal)
```

Временные решения помечаются специальным тегом с объяснением:

```python
# TODO(arch): децимация сделана через np.convolve для простоты;
# при записях > 10 млн отсчётов заменить на scipy.signal.decimate
# для корректной антиалиасинговой фильтрации.
```

## Правила внесения изменений

### Ветки

Каждое изменение разрабатывается в отдельной ветке. Имя ветки отражает суть изменения:

```
feature/add-hamming-window-option
fix/outlier-detection-nan-handling
refactor/split-psd-calculator-module
docs/update-algorithm-description
```

### Коммиты

Каждый коммит описывает одно логическое изменение. Сообщение коммита состоит из типа и краткого описания:

```
feat: add Hamming window option to spectral settings
fix: handle NaN in outlier detection when all values are zero
refactor: extract peak interpretation into separate function
test: add parametrized tests for Reynolds number edge cases
docs: clarify Strouhal table interpolation method
```

Типы: `feat` (новая функция), `fix` (исправление), `refactor` (рефакторинг без изменения поведения), `test` (тесты), `docs` (документация), `perf` (производительность), `build` (сборка).

### Code review

Все изменения в основную ветку проходят через pull request. Автоматические проверки (тесты, линтер, форматирование) должны пройти до проверки человеком.

При code review проверяется не только «работает ли код», но и:
- соответствует ли новый код архитектурным принципам из `02_architecture.md`;
- покрыто ли новое поведение тестами;
- обновлена ли документация, если изменилось публичное поведение.
