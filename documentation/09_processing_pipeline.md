# 09. Конвейер обработки данных — Industrial Vibration Analyzer

## Назначение документа

Этот документ описывает последовательность всех шагов обработки данных — от загрузки файла до формирования результата. Каждый шаг получает чётко определённые данные на входе и возвращает чётко определённые данные на выходе. Документ служит основным ориентиром при реализации `app/workflow_coordinator.py` и `app/analysis_runner.py`.

---

## Общая схема конвейера

```
[Файл на диске]
       │
       ▼
 ┌─────────────┐
 │  1. Чтение  │  infrastructure/readers/
 └──────┬──────┘
        │  RawFileData
        ▼
 ┌──────────────────┐
 │  2. Валидация    │  core/models/ + infrastructure/
 └────────┬─────────┘
          │  ValidatedSignalData
          ▼
 ┌──────────────────────┐
 │  3. Предобработка    │  core/signal/
 └──────────┬───────────┘
            │  ProcessedSignalData
            ▼
 ┌──────────────────────────┐
 │  4. Спектральный анализ  │  core/spectrum/
 └────────────┬─────────────┘
              │  SpectrumResult
              ▼
 ┌──────────────────────────┐
 │  5. Физические расчёты   │  core/physics/
 └────────────┬─────────────┘
              │  PhysicsResult
              ▼
 ┌──────────────────────────┐
 │  6. Оценка риска         │  core/physics/lock_in_risk.py
 └────────────┬─────────────┘

              ▼
 ┌──────────────────────────┐

 │  8. Обновление интерфейса│  ui/ (через сигналы Qt)
```

## Шаг 1 — Чтение файла

**Ответственный модуль:** `infrastructure/readers/`

**Вход:** путь к файлу, выбранному пользователем.

**Процесс:**
1. Определить формат файла по расширению (`.csv`, `.parquet`, `.xlsx`).
2. Выбрать подходящий ридер.
3. Прочитать файл в унифицированное промежуточное представление.
4. Заполнить объект `RawFileData`: таблица данных, имена колонок, строка метаданных файла.

**Выход:** объект `RawFileData`.

**Обработка ошибок:** если файл не читается (повреждён, неверный формат, нет прав доступа), поднимается исключение `FileReadError` с понятным сообщением. Чтение не продолжается.

**Ограничение:** при размере файла более 500 МБ выводится предупреждение до начала чтения.

---

## Шаг 2 — Валидация и подготовка данных

**Ответственный модуль:** `core/models/signal_data.py`, `infrastructure/`

**Вход:** `RawFileData` + назначения ролей для колонок + параметры, введённые пользователем.

**Процесс:**
1. Проверить монотонность временно́й оси.
2. Оценить постоянство шага дискретизации.
3. Подсчитать долю пропусков в каждой колонке.
4. Определить частоту дискретизации (из файла или от пользователя).
5. Сформировать перечень предупреждений.
6. Перевести физические единицы, если заданы коэффициенты преобразования.

**Выход:** объект `ValidatedSignalData` + список предупреждений `List[ValidationWarning]`.

**Обработка ошибок:** критические условия (немонотонная временна́я ось, все значения нулевые) приводят к `ValidationError`. Некритические условия формируют предупреждения, которые отображаются пользователю.

---

## Шаг 3 — Предобработка сигнала

**Ответственный модуль:** `core/signal/`

**Вход:** `ValidatedSignalData` + настройки предобработки из `AnalysisSettings`.

**Процесс (порядок строго зафиксирован):**

1. **Устранение постоянной составляющей** (`preprocessor.py`): вычитание среднего значения из всего сигнала. Выполняется всегда, отключить нельзя.

2. **Удаление выбросов** (`outlier_detector.py`): скользящий медианный порог. Параметры: ширина окна, пороговое значение k. Применяется, если включено пользователем.

3. **Заполнение пропусков** (`preprocessor.py`): линейная интерполяция для промежутков не длиннее порога; остальные промежутки заполняются нулями. Применяется, если включено пользователем.

4. **Применение полосового фильтра** (`filter.py`): фильтр Баттерворта с двойным проходом. Параметры: нижняя и верхняя граничные частоты, порядок фильтра. Применяется, если включено пользователем.

5. **Нормирование** (`preprocessor.py`): умножение на коэффициент преобразования датчика. Применяется, если коэффициент задан.

**Выход:** объект `ProcessedSignalData` с двумя версиями сигнала: исходный (после удаления выбросов и заполнения пропусков) и отфильтрованный.

**Важное правило:** порядок шагов изменить нельзя. Фильтрация до заполнения пропусков приводит к артефактам на границах промежутков. Устранение постоянной составляющей до фильтрации устраняет возможность сдвига нуля после фильтра.

## Шаг 4 — Спектральный анализ

**Ответственный модуль:** `core/spectrum/`

**Вход:** `ProcessedSignalData` (используется отфильтрованный сигнал) + настройки спектрального анализа.

**Процесс:**
1. **Вычисление PSD методом Уэлча** (`psd_calculator.py`): формирование сегментов, применение оконной функции, вычисление периодограмм, усреднение.
2. **Поиск пиков** (`peak_finder.py`): нахождение локальных максимумов выше порога, вычисление ширины на уровне -3 дБ, фильтрация шумовых пиков.
3. **Расчёт RMS** (`rms_calculator.py`): по всему сигналу, в заданной полосе частот, в скользящем временном окне.

**Выход:** объект `SpectrumResult` содержит: массив частот, массив PSD, список объектов `SpectralPeak`, значение общего RMS, массив скользящего RMS.

---

## Шаг 5 — Физические расчёты

**Ответственный модуль:** `core/physics/`

**Вход:** физические параметры из `AnalysisSettings` + `SpectrumResult`.

**Процесс:**
1. Расчёт числа Рейнольдса (`reynolds_calculator.py`).
2. Расчёт числа Струхаля (`strouhal_calculator.py`): для одиночного цилиндра по Re; для тандема — по таблице значений из диссертационного исследования.
3. Расчёт частоты срыва вихрей (`vortex_frequency.py`).
4. Расчёт приведённой скорости и отношения частот (`vortex_frequency.py`).

**Выход:** объект `PhysicsResult`.

---

## Шаг 6 — Оценка риска резонанса

**Ответственный модуль:** `core/physics/lock_in_risk.py`

**Вход:** `PhysicsResult` + `SpectrumResult`.

**Процесс:**
1. Вычисление относительного расстояния между расчётной частотой срыва и собственной частотой.
2. Проверка амплитуды ближайшего пика относительно порогового уровня.
3. Присвоение уровня риска: безопасный / наблюдение / критический.
4. Формирование текстовой рекомендации.

**Выход:** объект `RiskAssessment`.

---

## Шаг 7 — Сборка полного результата

**Ответственный модуль:** `app/analysis_runner.py`

**Вход:** все результаты предыдущих шагов.

**Процесс:** сборка единого объекта `AnalysisResult`, фиксация времени завершения расчёта, запись результирующей записи в журнал.

**Выход:** объект `AnalysisResult` — единственный объект, который получает интерфейс.

---

## Шаг 8 — Обновление интерфейса

**Ответственный модуль:** `ui/` (через механизм сигналов Qt)

`analysis_runner.py` испускает сигнал `analysis_completed(result: AnalysisResult)`. Каждая страница интерфейса подключена к этому сигналу и обновляет своё отображение при его получении. Страницы не знают о существовании других страниц.

---

## Конвейер сравнения с CFD

Если загружен второй файл (расчётные данные), параллельно основному конвейеру выполняется укороченный конвейер:

```
[CFD-файл] → Чтение → Валидация → [CfdSignalData]
                                         │
[ProcessedSignalData] ──────────────────►│
                                         ▼
                              core/validation/experiment_vs_cfd.py
                                         │
                              [ValidationResult]
```

Результат сравнения включается в общий `AnalysisResult`.

---

## Детерминированность конвейера

Конвейер является детерминированным: при одинаковых входных данных и одинаковых настройках результат всегда идентичен. Случайные числа в конвейере не используются. Это является обязательным требованием для научного инструмента.
