# 02. Архитектура приложения — Industrial Vibration Analyzer

## Назначение документа

Этот документ описывает структуру приложения: из каких частей оно состоит, как эти части взаимодействуют и почему принята именно такая организация. Документ является основным ориентиром для разработчика, который начинает работу с проектом, а также справочником при принятии решений об изменении кода.

---

## Ведущий принцип архитектуры

Архитектура IVA построена на принципе явного разделения ответственности. Каждый слой приложения делает строго одно дело и не знает о деталях реализации других слоёв. Взаимодействие между слоями происходит только через явно определённые интерфейсы.

Это решение принято по следующей причине: научное приложение имеет тенденцию накапливать логику в неожиданных местах — расчёты начинают «прорастать» в обработчики кнопок, форматирование данных смешивается с физическими вычислениями. Жёсткое разделение слоёв предотвращает этот сценарий с самого начала.

---

## Слои приложения

Приложение разделено на четыре слоя. Каждый слой зависит только от слоя, расположенного непосредственно под ним. Обратные зависимости запрещены.

```
┌─────────────────────────────────────────┐
│           Слой представления            │  ui/
│    (окна, виджеты, графики, формы)      │
└────────────────────┬────────────────────┘
                     │ вызывает
┌────────────────────▼────────────────────┐
│         Слой приложения                 │  app/
│  (координация, состояние, сценарии)     │
└────────────────────┬────────────────────┘
                     │ вызывает
┌────────────────────▼────────────────────┐
│          Слой предметной области        │  core/
│   (расчёты, алгоритмы, правила)         │
└────────────────────┬────────────────────┘
                     │ вызывает
┌────────────────────▼────────────────────┐
│         Слой инфраструктуры             │  infrastructure/
│  (файлы, форматы, логи, экспорт)        │
└─────────────────────────────────────────┘
```

### Слой представления (`ui/`)

Содержит всё, что пользователь видит и с чем взаимодействует: главное окно, вкладки, формы ввода параметров, графические виджеты, диалоговые окна. Этот слой не выполняет вычислений и не читает файлы. Его единственная задача — показывать данные и передавать действия пользователя в слой приложения.

### Слой приложения (`app/`)

Содержит координирующую логику: что происходит при нажатии кнопки «Запустить анализ», в каком порядке вызываются модули расчёта, как передаются данные между этапами обработки, как формируется состояние текущего сеанса работы. Этот слой не знает, как именно устроены расчёты и как именно выглядит интерфейс.

### Слой предметной области (`core/`)

Содержит всю научно-инженерную логику: алгоритмы предобработки сигналов, вычисление спектров, расчёт физических коэффициентов, оценку риска резонанса, сравнение данных. Этот слой полностью независим от интерфейса и инфраструктуры — его модули можно запускать, тестировать и использовать без запуска GUI.

### Слой инфраструктуры (`infrastructure/`)

Содержит всё, что связано с файловой системой и внешними форматами: чтение CSV, Parquet и Excel, запись PDF- и CSV-отчётов, ведение журнала событий, управление пользовательскими настройками. Этот слой не содержит бизнес-логики.

## Поток данных при выполнении анализа

Ниже описан путь данных от загрузки файла до отображения результата. Понимание этого пути необходимо каждому разработчику, работающему с проектом.

**Шаг 1 — Пользователь выбирает файл.** Страница импорта (`ui/pages/import_page.py`) получает путь к файлу и передаёт его в слой приложения.

**Шаг 2 — Координатор запускает чтение.** `workflow_coordinator.py` выбирает подходящий ридер из `infrastructure/readers/` и читает файл. Результат — объект типа `SignalData` (модель из `core/models/signal_data.py`).

**Шаг 3 — Данные сохраняются в сессии.** `analysis_session.py` хранит текущий `SignalData` и параметры, выбранные пользователем. Всё состояние сеанса находится в одном месте.

**Шаг 4 — Пользователь запускает анализ.** `analysis_runner.py` запускает обработку в фоновом потоке. Интерфейс отображает индикатор прогресса.

**Шаг 5 — Расчётный пайплайн.** Последовательно вызываются модули из `core/`: предобработка, спектральный анализ, физические расчёты, оценка риска. Каждый модуль получает данные на вход и возвращает результат. Промежуточные результаты накапливаются в объекте `AnalysisResult`.

**Шаг 6 — Результат передаётся в интерфейс.** После завершения расчётов `analysis_runner.py` отправляет сигнал в главное окно. Страницы обновляют отображение на основании нового `AnalysisResult`.

## Принципы, которые нельзя нарушать

**Нет циклических зависимостей.** Слой представления не импортируется из слоя предметной области. Слой предметной области не импортируется из слоя инфраструктуры. Нарушение этого правила разрушает изоляцию слоёв.

**Расчёты только в `core/`.** Любая формула, любое числовое преобразование, любое физическое правило находится только в директории `core/`. Если расчёт обнаружен в `ui/` или `app/` — это ошибка архитектуры, которую нужно исправить немедленно.

**Интерфейс не знает о файлах.** Страницы интерфейса не открывают файлы, не читают CSV и не пишут на диск. Всё это делает инфраструктурный слой.

**Одна точка входа для состояния.** Текущее состояние сеанса хранится только в `analysis_session.py`. Не допускается передача данных между страницами через глобальные переменные или атрибуты виджетов.

## Взаимодействие между слоями через сигналы Qt

Интерфейс PySide6 использует механизм сигналов и слотов для связи между компонентами. Это обеспечивает слабую связанность: отправитель не знает, кто слушает его сигнал.

Правило именования: сигналы описывают событие в прошедшем времени (`analysis_completed`, `file_loaded`, `error_occurred`). Слоты описывают реакцию в повелительном наклонении или инфинитиве (`update_overview_page`, `show_error_message`).
