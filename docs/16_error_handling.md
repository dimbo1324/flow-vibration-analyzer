# 16. Обработка ошибок — Industrial Vibration Analyzer

## Назначение документа

Этот документ описывает стратегию обработки ошибок: какие ошибки существуют, как они перехватываются, как сообщаются пользователю и разработчику, и какие принципы соблюдаются при написании кода обработки ошибок.

---

## Принципы обработки ошибок

**Ошибки ожидаются, а не игнорируются.** Любая внешняя операция (чтение файла, вычисление с пользовательскими данными) может завершиться неожиданно. Код, не обрабатывающий ошибки, является неполным кодом.

**Ошибки обрабатываются на нужном уровне.** Исключение перехватывается там, где есть достаточно контекста для принятия решения о дальнейших действиях. Исключение не должно «тихо» проглатываться в нижних слоях — оно поднимается вверх, пока не достигает уровня, где понятно, что делать.

**Пользователь видит причину, не детали.** Сообщение об ошибке для пользователя описывает проблему с его точки зрения («Файл не может быть прочитан: временна́я ось содержит нечисловые значения»), а не детали реализации («TypeError at csv_reader.py line 47»).

**Разработчик видит детали.** Технический стек вызовов записывается в журнал, но не отображается в интерфейсе.

**Частичная ошибка не останавливает всё.** Если один шаг пайплайна не может быть выполнен из-за отсутствия данных (например, физические расчёты не выполняются, потому что пользователь не ввёл параметры), остальные шаги выполняются, а результат в соответствующей части помечается как недоступный.

## Иерархия исключений

Все пользовательские исключения наследуются от базового `IVAError`:

```
IVAError
├─ FileReadError           # Файл не прочитан
│  ├─ FileNotFoundError    # Файл не существует
│  ├─ UnsupportedFormatError  # Формат не поддерживается
│  └─ FileCorruptedError   # Файл повреждён
│
├─ ValidationError         # Данные не прошли проверку
│  ├─ NonMonotonicTimeAxisError
│  ├─ EmptySignalError
│  └─ InsufficientDataError  # Слишком мало данных для анализа
│
├─ ProcessingError         # Ошибка в расчётном модуле
│  ├─ FilterConfigurationError  # Неверные параметры фильтра
│  └─ SpectrumCalculationError
│
├─ PhysicsInputError       # Недопустимые физические параметры
└─ ExportError             # Ошибка при сохранении файла
```

Каждое исключение содержит:
- `user_message: str` — сообщение для отображения пользователю;
- `technical_details: str` — технические детали для журнала;
- `recovery_hint: str | None` — подсказка о том, что пользователь может сделать.

## Обработка ошибок по слоям

### Инфраструктурный слой (`infrastructure/`)

Все исключения Python, возникающие при чтении файлов (`OSError`, `UnicodeDecodeError`, `pandas.errors.ParserError` и т.д.), перехватываются и преобразуются в соответствующие `IVAError`.

```python
def read_csv(file_path: str) -> RawFileData:
    try:
        df = pandas.read_csv(file_path, ...)
    except FileNotFoundError:
        raise FileNotFoundError(
            user_message=f"Файл не найден: {Path(file_path).name}",
            technical_details=f"Full path: {file_path}"
        )
    except UnicodeDecodeError as e:
        raise FileCorruptedError(
            user_message="Не удалось прочитать файл: возможно, он сохранён в неподдерживаемой кодировке",
            technical_details=str(e),
            recovery_hint="Попробуйте пересохранить файл в кодировке UTF-8"
        )
```

### Слой предметной области (`core/`)

Расчётные модули проверяют входные данные в начале функции и поднимают специфичные исключения при нарушении контракта:

```python
def apply_bandpass_filter(signal, sampling_rate_hz, low_hz, high_hz, order):
    nyquist = sampling_rate_hz / 2
    if not (0 < low_hz < high_hz < nyquist):
        raise FilterConfigurationError(
            user_message=f"Граничные частоты фильтра выходят за допустимый диапазон 0–{nyquist} Гц",
            recovery_hint=f"Допустимые значения: от 0 до {nyquist} Гц"
        )
    ...
```

Математические ошибки (`numpy.linalg.LinAlgError`, деление на ноль) перехватываются и преобразуются в `ProcessingError`.

### Слой приложения (`app/`)

`analysis_runner.py` перехватывает все `IVAError` от нижних слоёв, записывает их в журнал и испускает сигнал `error_occurred(error: IVAError)` для интерфейса.

Необработанные исключения (не являющиеся `IVAError`) перехватываются как `CRITICAL` и приводят к аварийному дампу (см. `15_logging_and_audit.md`).

### Слой представления (`ui/`)

Получив сигнал `error_occurred`, интерфейс отображает пользователю `error.user_message` в виде баннера с кнопкой «Подробнее». По кнопке «Подробнее» открывается диалог с `error.technical_details` и `error.recovery_hint`.

## Специальные случаи

### Отсутствие необязательных параметров

Если пользователь не ввёл параметры течения, физические расчёты не выполняются. `AnalysisResult.physics` равен `None`. Страница «Физика» отображает сообщение: «Параметры течения не заданы. Перейдите на вкладку "Физика" для ввода параметров». Это не ошибка — это допустимое состояние.

### Прерывание анализа пользователем

Если пользователь нажимает «Отменить» в процессе анализа, рабочий поток останавливается чисто: текущие промежуточные результаты сохраняются не частично, состояние сессии возвращается к «готов к запуску». Прерывание фиксируется в журнале как `INFO`.

### Ошибка при сохранении отчёта

Если нет прав на запись в папку назначения или не хватает места на диске, выводится диалог с предложением выбрать другую папку. Уже вычисленный результат при этом не теряется.

---

## Что никогда не должно происходить

- Приложение не должно аварийно завершаться без отображения понятного сообщения пользователю.
- Пустой `except:` без записи в журнал и переброса недопустим.
- Исключение не должно быть заглушено (`pass`) без явного комментария о причине.
- Сообщение пользователю не должно содержать имена файлов исходного кода, номера строк или названия классов исключений Python.
